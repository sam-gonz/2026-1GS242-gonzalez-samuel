Daily Scrum · Pregunta 1

¿Qué hicimos desde el último laboratorio?

Avances concretos del proyecto DDS enfocados en desarrollo, medición y soporte a la trazabilidad.

  • Avance del desarrollo: refinamos el formulario DDS, consolidamos requisitos RF/RNF y actualizamos diagramas de casos de uso y clases.
  • Documento de métricas: definimos las métricas de Unidad II (requerimientos, diseño, código y métricas clásicas) y su proceso de medición.
  • Registro de defectos y pruebas: actualizamos la matriz de defectos simulados y los casos de prueba alfa/beta vinculados al DDS.
Evidencias del avance

Repositorio GitHub del proyecto con la documentación de debida diligencia, el diseño de requisitos y la presentación web interactiva para exponer las métricas del Parcial 2.

Daily Scrum · Pregunta 2

¿Qué haremos para el próximo laboratorio?

Plan de trabajo para la siguiente iteración, organizado por entregables semanales y categorías de métricas.

1. Avance del desarrollo

Completar casos de uso pendientes, refinar la lógica de clasificación de riesgo y alinear el código con los diagramas actualizados.

2. Evidencias de medición

Aplicar métricas ISTQB (defectos, cobertura, esfuerzo), TMMi (control y mejora) y TQM (enfoque de proceso) sobre las pruebas y el flujo DDS.

3. Registro de métricas

Levantar valores en las cinco categorías: requerimientos, diseño, código, métricas clásicas y proceso de medición documentado.

4. Herramientas y backlog

Actualizar tablero Kanban (historias en progreso, bloqueadas y completadas) y generar reportes desde GitHub y herramientas de pruebas/sonar.

Daily Scrum · Pregunta 3

¿Qué problemas o bloqueos tuvimos?

Riesgos e impedimentos que pueden afectar la medición y la calidad del proyecto si no se gestionan.

  • Configuración de herramientas: esfuerzo adicional para levantar y conectar SonarQube y herramientas de pruebas al repositorio.
  • Datos para métricas: necesidad de ejecutar suficientes pruebas para que las métricas ISTQB y TMMi sean significativas.
  • Carga de trabajo: balancear desarrollo funcional, documentación de métricas y actualización del backlog dentro del tiempo de laboratorio.
Acciones de mitigación

Priorizar la automatización de pruebas clave, agendar tiempo específico para configurar herramientas de análisis y mantener visible en el tablero Kanban cualquier bloqueo técnico o dependencia externa.

Código

Densidad de defectos, code smells, vulnerabilidades y debt ratio obtenidos con análisis estático del código.

Proceso

Métricas clásicas y ciclo de medición

KLOC, productividad y tasa de defectos, documentando siempre objetivo, definición, recolección, análisis y acción de mejora de cada métrica.
